# 管理员端/教师端前端结构设计

> 日期：2026-04-26
> 适用范围：`frontend/apps/admin-app/`、`backend/ckqa-back/`、`pdf_ingest/`、`graphrag_pipeline/`
> 目标：为管理员端与教师端共用平台确定前端信息架构、布局结构、页面数量与首版落地范围。本设计只涉及结构，不涉及页面视觉风格、配色、动效与组件美术规范。

> 实现状态更新：截至 2026-04-29，`frontend/apps/admin-app/` 已经完成核心运维页的 Java `/api/v1` 真实接入，覆盖系统健康、课程、资料生命周期、知识库列表/详情、构建向导、索引详情和 QA 冒烟验证；详细真实数据接入设计稿与实施计划已归档到 `docs/superpowers/archive/`。

## 1. 定位

管理员端与教师端建议合并为同一个前端平台，定位为“课程知识库构建与运维平台”。

平台核心任务不是通用教务管理，而是围绕 CKQA 当前主链路完成知识库生产闭环：

```text
课程
  -> 课程资料
  -> PDF 解析
  -> GraphRAG 输入导出
  -> 知识库索引构建
  -> 索引激活
  -> 问答验证
  -> 检索日志与审计
```

管理员与教师不拆成两个应用，而是通过 RBAC 控制可见菜单、可操作按钮和可访问数据范围：

| 角色 | 典型职责 | 默认数据范围 |
| --- | --- | --- |
| 平台管理员 | 维护全局课程、用户、角色、权限、系统状态与审计记录 | 全部课程与全部用户 |
| 教师 | 管理自己负责课程的资料、知识库、索引与问答运行情况 | 自己作为 `teacher` / `assistant` 成员的课程 |
| 助教 | 协助教师维护资料、查看任务与日志 | 被授权课程 |
| 只读运维/审计角色 | 查看系统状态、任务日志、授权审计 | 按权限点配置 |

## 2. 设计边界

### 2.1 本结构覆盖

1. 管理员/教师共用的后台导航结构。
2. 课程、资料、知识库、索引、问答、权限、审计的页面清单。
3. 路由组织建议。
4. RBAC 对菜单、按钮和数据范围的影响。
5. 首版 MVP 页面范围。

### 2.2 本结构不覆盖

1. 页面视觉风格、配色、字体、插画、动效。
2. 具体组件库选型与组件样式规范。
3. 复杂教务系统能力，例如排课、班级、选课审批、成绩管理。
4. 学习进度、收藏、反馈、社区运营等学员端能力。
5. 后端接口新增方案；本设计默认优先复用 Java `/api/v1` 业务入口。

## 4. 总体布局结构

建议使用四类布局壳。

| 布局 | 适用页面 | 结构 |
| --- | --- | --- |
| `AuthLayout` | 登录、无权限、错误页 | 居中内容区，可承载登录表单或状态提示 |
| `ConsoleLayout` | 后台主页面 | 顶部栏 + 左侧一级导航 + 主内容区 |
| `DetailLayout` | 课程、资料、知识库、用户详情 | 顶部标题区 + 基础信息区 + Tabs 内容区 |
| `WorkflowLayout` | 知识库构建向导 | 步骤条 + 当前步骤面板 + 任务状态/日志区域 |

### 4.1 `ConsoleLayout`

主后台布局：

```text
顶部栏
  - 平台名称
  - 当前用户
  - 当前角色/数据范围
  - 系统健康入口
  - 退出登录

左侧导航
  - 工作台
  - 课程与资料
  - 知识库构建
  - 问答运维
  - 用户与权限
  - 系统与审计

主内容区
  - 面包屑
  - 页面标题与主操作
  - 页面主体
```

### 4.2 `DetailLayout`

详情页统一采用同一结构，减少课程、资料、知识库、用户详情页的割裂感：

```text
标题区
  - 主标题
  - 状态标签
  - 关键操作按钮

摘要区
  - 核心字段
  - 最近更新时间
  - 所属课程/知识库/用户等关联对象

Tabs
  - 概览
  - 资料/文档/成员/任务/日志等具体内容
```

### 4.3 `WorkflowLayout`

知识库构建流程建议用向导页承载，不拆散到过多孤立页面：

```text
步骤条
  1. 选择课程资料
  2. 解析状态检查
  3. 导出 GraphRAG 输入
  4. 创建索引运行
  5. 激活索引版本
  6. 问答冒烟验证

内容区
  - 当前步骤表单/状态/结果

侧边或底部任务区
  - 最近任务
  - 错误信息
  - 日志入口
```

  /roles [MVP]

建议路由元信息至少包含：

```js
{

  layout: 'console',
  permissions: ['course:read'],
  status: 'mvp',
  hidden: false,
  keepAlive: true
}
```

详情页可增加资源类型和数据范围检查信息：

```js
{
  title: '课程详情',
  layout: 'detail',
  permissions: ['course:read'],
  status: 'mvp',
  resource: 'course',
  scope: 'course'
}
```

后续页面应显式标记状态，避免实现时误认为已经纳入首版范围：

```js
{
  title: '索引运行列表',
  layout: 'detail',
  permissions: ['kb:read'],
  status: 'upcoming',
  routeState: 'coming-soon'
}
```

### 7.1 认证与路由守卫基线

认证层是第一阶段结构骨架的一部分，不应只作为后续遗留项处理。即使后端正式登录接口尚未确定，前端也需要先建立稳定的认证抽象：

1. `AuthStore`：保存当前用户、角色、权限点、课程数据范围和登录状态。
2. `token` 策略：明确 token 存储位置、过期处理和退出登录清理逻辑。首版可先用开发态 mock token，但接口形状要与真实登录保持一致。
3. 路由白名单：至少包含 `/login`、`/403`、`/404`、`/500`。
4. 路由守卫：未登录用户访问业务页跳转 `/login`；已登录但缺少权限跳转 `/403`。
5. Axios 拦截器：统一追加认证头，处理 `401`、`403` 和 Java `ApiResponse` 错误信息。
6. 开发态身份切换：在真实登录接口完成前，可提供管理员/教师两种本地 mock 身份，用于验证菜单、按钮和数据范围。

## 8. 页面职责

### 8.1 工作台总览

目标：让管理员/教师一进入平台就知道当前知识库生产链路是否健康。

建议展示：

1. 当前用户可管理课程数量。
2. 资料总数与解析状态分布。
3. 知识库数量与激活索引状态。
4. 最近索引运行。
5. 最近问答任务失败/超时情况。
6. 系统健康摘要。

### 8.2 系统健康

目标：承接 `GET /api/v1/system/health` 的可视化展示。

建议展示：

1. MySQL 状态。
2. `pdf_ingest` 根目录状态。
3. `graphrag_pipeline` 根目录状态。
4. GraphRAG API 连通性。
5. GraphRAG 输出目录与 LanceDB 是否 ready。
6. 每个子项的 `reachable`、`ready`、`message`。

### 8.3 课程列表

目标：作为教师/管理员进入知识库构建链路的主入口。

建议功能：

1. 按课程名称、课程 ID、状态筛选。
2. 展示课程资料数量、知识库数量、最近索引状态。
3. 管理员可新建/编辑课程。
4. 教师只看到自己有课程成员关系的课程。

### 8.4 课程详情

目标：汇总课程下的资料、知识库、成员和问答状态。

建议 Tabs：

1. 概览
2. 课程资料
3. 知识库
4. 课程成员
5. 问答会话
6. 操作日志

### 8.5 资料详情

目标：查看一份课程资料在当前课程上下文中的生命周期。

建议展示：

1. 课程资料 ID。
2. 资料对象 ID。
3. 文件名、大小、MD5。
4. 资料类型。
5. 解析状态。
6. MinerU 批次 ID。
7. 解析开始/完成时间。
8. 错误信息。

建议操作：

1. 触发解析。
2. 查看解析结果。
3. 导出 GraphRAG 输入。
4. 跳转到所属知识库构建向导。

### 8.6 解析结果详情

目标：查看 MinerU 和标准化导出产物。

建议展示：

1. `content_list_json`
2. `model_json`
3. `layout_json`
4. `markdown`
5. `origin_pdf`
6. GraphRAG 投影产物入口

首版可以只做只读列表与下载/查看链接，不需要内置复杂 JSON 编辑器。

### 8.7 知识库列表

目标：管理课程知识库实例。

建议展示：

1. 知识库名称。
2. 所属课程。
3. 状态：`draft` / `active` / `archived`。
4. 当前激活索引运行 ID。
5. 最近索引状态。

### 8.8 知识库详情

目标：查看知识库当前状态与构建入口。

建议 Tabs：

1. 概览
2. 文档映射
3. 索引运行
4. 问答验证
5. 运行日志

建议主操作：

1. 进入构建向导。
2. 创建索引运行。
3. 激活成功索引。
4. 归档知识库。

### 8.9 构建向导

目标：把复杂知识库构建过程收敛到一个连续流程。

步骤建议：

| 步骤 | 内容 | 关联动作 |
| --- | --- | --- |
| 1. 选择课程资料 | 选择已解析或待解析资料 | 读取课程资料列表 |
| 2. 解析状态检查 | 判断资料是否 `done` | 必要时触发解析 |
| 3. 导出输入 | 生成 `section_docs.json` / `page_docs.json` | 调用导出接口 |
| 4. 创建索引 | 触发 GraphRAG index | 创建 `index_runs` |
| 5. 激活索引 | 设置当前可用索引版本 | 更新知识库激活索引 |
| 6. 问答验证 | 用 `basic` 或 `local` 做冒烟验证 | 创建会话并提交问题 |

向导不是严格线性表单，而是“带前置条件的可跳步流程”：

1. 每一步都应有状态：`todo` / `running` / `done` / `skipped` / `blocked` / `failed`。
2. 已解析资料、已导出输入、已有成功索引时，对应步骤应显示为 `done` 或 `skipped`，允许用户直接进入后续步骤。
3. 不能绕过真实前置条件，例如资料未解析成功时不能创建有效索引。
4. 已完成步骤允许回看和重跑；重跑时需要展示是否覆盖已有结果、是否触发 `force` 类操作。
5. 向导页负责展示流程状态，不应隐藏真实失败日志；失败详情仍应能跳转到解析结果、索引运行或检索日志页面。

问答验证步骤产生的会话应与正式用户问答区分：

1. 验证会话会进入 `/app/qa-sessions`，因为它是真实调用链路的一部分。
2. 前端列表应提供“冒烟验证/正式问答”过滤，避免教师把验证会话误认为正式学习会话。
3. 若后端暂未提供 `session_type` 字段，首版可以用标题前缀、创建来源标签或前端约定字段临时标记；后续建议补充持久化字段。
4. 验证会话默认不进入普通统计看板，除非用户显式选择包含测试数据。

### 8.10 索引运行详情

目标：让管理员/教师理解一次索引任务是否成功、失败在哪里。

建议展示：

1. 知识库 ID。
2. 引擎。
3. 索引版本。
4. 状态。
5. 开始/结束时间。
6. 运行元数据。
7. 索引产物。
8. 失败信息与重试入口。

### 8.11 问答会话列表

目标：查看课程知识库上线后的问答使用情况。

建议展示：

1. 会话标题。
2. 用户。
3. 课程。
4. 知识库。
5. 状态。

## 9. RBAC 设计

### 9.1 控制层级

前端 RBAC 建议分三层：

| 层级 | 控制内容 | 示例 |
| --- | --- | --- |
| 菜单权限 | 是否显示导航入口 | 没有 `user:read` 不显示“用户与权限” |
| 操作权限 | 是否显示按钮和执行动作 | 没有 `kb:index` 不显示“创建索引” |
| 数据范围 | 是否能访问具体资源 | 教师只能访问自己课程 |

### 9.2 首版权限点

建议首版权限点如下：

```text
course:read
course:write

material:read
material:upload
material:parse
material:export

kb:read
kb:write
kb:index
kb:activate

qa:read
qa:log:read

membership:read
membership:write

user:read
user:write

role:read
role:write

audit:read
system:read
```

其中 `material:upload` 是面向后续正式上传链路的预留权限点。Java 侧上传入口未确认前，首版只保留权限字段和路由扩展位，不实现资料上传按钮或上传表单。

### 9.3 默认角色建议

| 角色 | 权限建议 |
| --- | --- |
| `admin` | 拥有全部权限 |
| `teacher` | 课程、资料、知识库、索引、问答运维相关权限；数据范围限制在授权课程 |
| `assistant` | 资料查看、解析、导出、问答日志查看；默认不授予 `kb:index`，如果后续需要按课程开放建索引能力，应补充课程级策略或成员权限覆盖模型 |
| `auditor` | 只读查看系统健康、问答日志、授权审计 |

如果后端当前只预置 `user` / `admin`，前端可以先按 `admin` 与“非 admin 用户”两档实现，菜单与按钮层保留权限点结构，等后端补齐角色后再细化。

## 12. 验证标准

结构实现后，建议按以下标准验收：

1. 管理员登录后能看到全部一级导航。
2. 教师登录后只能看到与自己课程相关的管理入口。
3. 课程详情能进入资料、知识库、成员与问答相关区域。
4. 知识库构建向导能表达从资料检查到索引激活的完整流程。
5. 系统健康页只调用 Java `/api/v1/system/health`。
6. 前端没有直接调用 GraphRAG Python `/v1` 的正式业务代码。
7. 未实现页面使用明确的“未开放”状态页，而不是空白页。
8. 路由元信息中保留权限字段，为后续真实 RBAC 接入留出口。
9. `/app/system` 是“系统与审计”的一级导航入口，首版默认进入 `/app/health`；工作台只展示健康摘要和跳转入口。
10. 后续页面在路由元信息中使用 `status: 'upcoming'`，不会被误纳入首版验收。
11. 课程成员页面使用 `membership:read` / `membership:write` 控制访问与操作。
12. 构建向导能显示 `done` / `skipped` / `blocked` 等步骤状态。
13. 问答冒烟验证会话在列表中可过滤，不混入正式会话统计。
14. 未登录、登录过期、无权限三种情况分别能进入 `/login`、重新登录流程和 `/403`。

## 13. 当前遗留问题

1. `backend/ckqa-back` 当前已覆盖多张标准表的基础控制器，但部分管理端需要的聚合接口可能仍需后续补齐，例如工作台聚合、课程详情聚合、角色权限矩阵批量保存。
2. Java 侧上传链路仍不是完整正式入口，当前课程资料链路主要继承 `pdf_ingest` 与已有数据库记录。
3. 上传入口和 `material:upload` 权限点在 Java 上传链路确认前暂不实施，首版不做资料上传 UI，避免前端入口与后端能力不匹配。
4. 后端角色预置目前偏最小化，前端可以先按权限点设计结构，但真实角色粒度需要后端进一步确认。
5. 问答检索命中 `qa_retrieval_hits` 当前仍属于后续增强点，前端首版应允许该区域为空。
6. 后端正式登录接口仍需确认；前端第一阶段先建立认证抽象和开发态 mock 身份，避免后续路由守卫、请求层和 RBAC 重写。
7. 问答验证会话目前缺少稳定的 `session_type` 字段；首版可用前端约定标记，后续建议补充后端持久化字段。
8. 助教是否可以按课程获得建索引能力目前没有独立策略模型支撑；如需实现，应在后端补充课程级策略或成员权限覆盖表。

## 14. 下一步建议

当前结构骨架和视觉重构已经完成，下一步建议从“结构建设”切换到“真实业务接入与验收”：

1. 优先接入 Java `/api/v1` 的真实数据页：`/app/courses`、`/app/knowledge-bases`、`/app/qa-sessions`。
2. 将工作台指标和近期任务从本地数组切换为 Java 聚合接口返回，保持 `DataSourceChip` 从 `mock` 向 `live` 渐进迁移。
3. 为 403 页面补精确缺失权限提示：路由守卫跳转时附带 `required` 查询参数。
4. 为控制台补浏览器级冒烟测试，覆盖登录、工作台、健康页、构建向导和未开放状态页。
5. 在后端聚合接口稳定后，再逐步补用户、角色权限矩阵、课程成员和授权审计日志等管理页的真实数据。
