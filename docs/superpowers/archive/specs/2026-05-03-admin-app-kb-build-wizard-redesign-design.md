# admin-app 知识库构建向导页面逻辑重设计稿

> 归档说明：该设计稿已在 `frontend/apps/admin-app/` 的知识库构建向导实现中落地，当前活跃入口见 `frontend/apps/admin-app/README.md`。
> 当前代码以 `frontend/apps/admin-app/src/views/pages/module-page-model.js`、`module-loaders.js`、`module-content.js`、`ModulePage.vue` 和 `src/components/build-wizard/` 为准。

> 日期：2026-05-03
> 适用范围：`frontend/apps/admin-app/`
> 相关路由：`/app/knowledge-bases/:kbId/build`
> 目标：重新组织知识库构建向导，让当前步骤成为页面主视觉与主交互区域，进度、状态、回退、前进和操作反馈都围绕当前步骤展开。

## 1. 背景与当前问题

当前构建向导已经接入 Java `/api/v1`，并能通过 loader 得到知识库、课程资料、索引运行、资料选择和问答效果验证所需数据。现有实现的主要结构是：

1. 顶部 `WorkflowStepper` 三列区域：左侧步骤列表，中间“当前动作”小面板，右侧任务状态列表。
2. 下方再用三块面板分别展示“本次构建资料”“索引运行”“问答效果验证”。
3. 当前步骤通过 `activeStepKey` 控制，但用户主要操作仍散落在下方不同面板。
4. 进度主要靠每个步骤的状态徽标表达，没有一个清晰的整体进度轨。
5. 页面没有显式“上一步 / 下一步”，用户能点步骤，但缺少连续向导的方向感。

这会导致几个体验问题：

1. 当前步骤没有占据页面主导位置，用户需要在步骤条、当前动作、任务状态和下方业务面板之间来回扫视。
2. “当前动作”只是一块小区域，无法承载每一步实际需要展示的资料、校验结果、日志和操作反馈。
3. 索引构建与问答效果验证虽然有局部反馈，但它们和步骤导航之间的关系不够强。
4. 缺少明确进度表达，用户很难一眼判断当前处在整个构建链路的百分之几、还剩哪些阻塞项。
5. 回看上一步、进入下一步、跳到可执行步骤这些行为缺少统一规则。

## 2. 设计目标

新的页面逻辑要满足：

1. 当前步骤在页面中间占据主导区域，每一步都能展示足够的信息、操作和状态。
2. 顶部使用整体进度轨表达完成度，进度条结合步骤状态，而不是只依赖小徽标。
3. 明确支持标题图标回退、业务主操作和步骤跳转，但跳转只改变查看位置，不绕过真实前置条件。
4. 每一步都能独立说明：当前状态、前置条件、已取得证据、可执行动作、阻塞原因、相关日志入口。
5. 保留现有 Java `/api/v1` 边界，不引入浏览器直接访问 GraphRAG Python `/v1`。
6. 首轮实现应尽量复用现有 loader、API 和状态映射，不新增后端契约。
7. 页面应像运维控制台，而不是营销页：信息密度要高，但主次清楚。
8. 视觉型短标签可以保留英文，例如 `PROCESS`、`NOTES`、步骤下方的 `Select material` / `Build index`；业务含义不直观的信息必须用中文，例如“知识库：主知识库”“已选资料：0 个”“可用索引：暂无”。

## 3. 方案比较

### 方案 A：主舞台式向导

结构：

1. 顶部是横向进度轨，展示 `完成数 / 总步骤数`、百分比、阻塞数、运行中任务。
2. 中间是单一主舞台，只渲染当前步骤的完整内容。
3. 当前步骤标题左侧提供图标回退，底部固定当前步骤业务动作区：主操作、刷新、辅助操作。
4. 右侧或下方使用可折叠诊断区展示最近任务、接口来源、日志入口。

优点：

1. 最贴合“当前动作和状态占据页面主导”的要求。
2. 改动集中在 `WorkflowStepper`、构建页分支和模型函数，风险较低。
3. 能继续复用当前真实状态模型，并扩展为六步流程。

缺点：

1. 一次只能展开一个步骤，需要通过进度轨切换查看其他步骤。

### 方案 B：纵向时间线全展开

结构：

1. 所有步骤在页面中从上到下展开，每个步骤都是一个大区块。
2. 当前步骤自动滚动到中间，并用高亮标记。
3. 标题旁仍有回退入口，底部仍使用业务主操作。

优点：

1. 历史步骤和后续步骤都可见，便于审计。
2. 适合长日志、长表格。

缺点：

1. 页面会很长，当前步骤不一定保持主导。
2. 需要处理滚动定位、吸顶导航和移动端高度，首轮复杂度更高。

### 方案 C：每一步独立子路由

结构：

1. `/build/material`、`/build/parse`、`/build/export` 等独立路由。
2. 每个路由专注一个步骤。

优点：

1. URL 可分享，浏览器历史天然支持回退。
2. 每一步页面可以非常完整。

缺点：

1. 路由、权限、面包屑、测试和状态恢复成本较高。
2. 对当前架构改动偏大，不符合“先最小可运行”的方向。

推荐采用方案 A：主舞台式向导。它能最大程度解决当前不舒服的布局问题，同时保持实现范围可控。

## 4. 新页面结构

```text
页面标题区
  - 知识库名称 / 状态 / 数据来源
  - 最近刷新时间 / 返回知识库详情

构建进度轨
  - 进度条：已完成步骤权重 / 总步骤权重
  - 状态摘要：已完成、可执行、执行中、失败、未满足条件的数量
  - 可点击步骤节点：资料、解析、导出、提示词调优、索引、问答效果验证

当前步骤主舞台
  - 左上：步骤序号、步骤名称、状态徽标、当前结论
  - 中部：该步骤专属信息区
  - 右侧或下方：证据、前置条件、阻塞原因
  - 内嵌局部反馈：运行中、确认中、成功、失败

步骤导航与操作区
  - 标题左侧图标回退
  - 当前步骤业务主操作
  - 刷新状态

诊断与日志区
  - 最近索引运行
  - 最近问答效果验证会话
  - 原始接口状态折叠区
```

核心变化：原本散落在下方的“资料选择 / 索引运行 / 问答效果验证”不再作为三个并列面板常驻，而是进入对应步骤的主舞台内容。

## 5. 步骤定义

首轮采用六步流程：在“导出图谱输入”之后补充“提示词调优”，再进入“创建索引”。提示词调优是降低课程语境偏差、抽取误差和后续索引质量风险的关键步骤，不能被折叠到索引动作里。

不再单独展示“切换可用索引”步骤。现有 Java 后端在 `POST /api/v1/knowledge-bases/{id}/index-runs` 成功后会自动回写当前可用索引；页面第 5 步只命名为“创建索引”，可用状态作为该步骤内部结果信息展示。

| 序号 | key | 名称 | 主舞台重点 | 默认主操作 |
| --- | --- | --- | --- | --- |
| 1 | `material` | 选择课程资料 | 可用资料表格、批量选择摘要、资料解析与导出聚合摘要 | 确认勾选 |
| 2 | `parse` | 解析状态检查 | 已选资料的解析任务列表、并行进度、解析状态 | 并行解析未完成资料 / 检查图谱输入 |
| 3 | `export` | 导出图谱输入 | 已选资料的标准化文档、章节输入、分页输入完整性 | 导出缺失输入 / 确认图谱输入 |
| 4 | `prompt` | 提示词调优 | 当前活动提示词说明、后续可接入的候选与评测摘要 | 确认提示词策略 / 进入创建索引 |
| 5 | `index` | 创建索引 | 最新索引运行、运行状态、当前可用索引 | 开始构建索引 / 进入问答验证 |
| 6 | `qa_check` | 问答效果验证 | 验证问题、回答摘要、会话链接、失败详情 | 发起问答验证 / 重新验证效果 |

当前不提前设计一个无法真实执行的独立切换按钮。

上表的“默认主操作”只是入口提示，真实按钮必须随子状态变化。首轮按以下映射收口，避免实现时各自理解：

| 步骤 | 子状态 | 主操作 |
| --- | --- | --- |
| 选择课程资料 | 未选择任何资料 | 确认勾选，禁用 |
| 选择课程资料 | 已选择且存在待解析 / 解析中 / 解析失败资料 | 确认勾选，成功后进入第 2 步 |
| 选择课程资料 | 已选择且全部解析完成 | 确认勾选，成功后直接进入第 3 步 |
| 解析状态检查 | 未选择资料 | 禁用，提示回到第 1 步 |
| 解析状态检查 | 任一已选资料解析中 | 刷新解析进度 |
| 解析状态检查 | 存在已选资料待解析或解析失败 | 并行解析未完成资料 |
| 解析状态检查 | 全部已选资料解析完成 | 检查图谱输入 |
| 导出图谱输入 | 任一已选资料解析中 | 禁用，提示等待解析完成 |
| 导出图谱输入 | 存在已选资料解析失败 | 禁用，提示回到第 2 步处理失败 |
| 导出图谱输入 | 已选资料全部解析完成但存在产物缺失 | 导出缺失输入 |
| 导出图谱输入 | 已选资料产物全部完整但尚未确认 | 确认图谱输入 |
| 导出图谱输入 | 已选资料产物全部完整且已确认 | 进入提示词调优 |
| 提示词调优 | 图谱输入未导出 | 禁用，提示先完成第 3 步 |
| 提示词调优 | 图谱输入已确认但尚未确认提示词策略 | 确认提示词策略 |
| 提示词调优 | `promptConfirmed=1` | 进入创建索引 |
| 创建索引 | 图谱输入不完整 | 禁用 |
| 创建索引 | 索引运行中 | 刷新状态 |
| 创建索引 | 无运行或最近运行失败 | 开始构建索引 |
| 创建索引 | 最近运行成功但 `activeIndexRunId` 暂未指向该运行 | 刷新可用状态 |
| 创建索引 | 索引已可用 | 进入问答验证 |
| 问答效果验证 | 缺少可用索引 | 禁用 |
| 问答效果验证 | 有可用索引但未验证 | 发起问答验证 |
| 问答效果验证 | 验证完成 | 重新验证效果 |

### 5.1 提示词调优的数据来源边界

已核对当前代码：`backend/ckqa-back` 的 Java `/api/v1` 暂时没有提示词调优状态端点，`frontend/apps/admin-app` 构建页 loader 也只读取知识库详情、课程资料、解析结果和索引运行。GraphRAG 侧真实调优链路存在于 `graphrag_pipeline/` 的脚本与文件中，例如 `prompts/final/active_prompt.json`、候选 Prompt 目录和评测报告，但这些文件当前没有通过 Java 后端暴露给浏览器。

因此在“不新增后端接口”的首轮范围内，第 4 步不能伪装成实时展示候选 Prompt、课程语境样本和评测结果。它应作为可见的“提示词调优确认”节点：

1. 图谱输入未导出或未确认时显示“未满足条件”。
2. 图谱输入已确认后显示“当前版本沿用 GraphRAG 已配置的活动提示词”，并说明候选详情、抽取评测和启用记录待后续 Java 接口接入。
3. “确认提示词策略”只把确认态写入 URL 查询参数 `promptConfirmed=1`，不写回服务端，也不把不存在的调优结果当成业务事实。
4. 首轮创建索引的硬前置仍以现有 Java 能判断的数据为准：GraphRAG 输入产物完整、索引接口可创建运行、没有进行中的重复任务。
5. 后续如果要让提示词调优成为真实阻塞项，应先由 Java `/api/v1` 暴露活动提示词、候选、评测和确认状态，再把第 4 步升级为后端可校验步骤。

## 6. 进度表达

进度不只看当前步骤序号，而是看步骤状态：

面向用户的状态必须使用中文：

| 内部状态值 | 用户可见文案 | 进度含义 |
| --- | --- | --- |
| `done` | 已完成 | 计入已完成 |
| `running` | 执行中 | 不计入完成，显示动态状态 |
| `failed` | 失败 | 不计入完成，显示失败标记 |
| `blocked` | 未满足条件 | 不计入完成，提示阻塞原因 |
| `ready` | 可执行 | 不计入完成，表示当前可操作 |

首轮不引入 `skipped` 状态，因为当前六步流程没有明确的跳过入口和可审计原因。后续如果允许跳过某一步，必须补充触发场景、权限、记录位置和对进度的影响。

状态徽标属于业务判断信息，应使用中文。步骤节点下方的短辅助语属于视觉导航信息，可以使用简洁英文，以避免和步骤标题重复。

进度公式：

```text
progress = 完成步骤数 / 总步骤数
```

首版所有步骤权重相同。后续如果要体现 GraphRAG 索引耗时更长，可以给 `index` 更高权重，但不建议首轮引入复杂加权。

首轮进度计数必须遵守显式确认语义：

1. 第 1 步只有在 `materialConfirmed=1` 且当前资料集合仍有效时计为“已完成”；当前资料集合可以由 `materialIds` 或有效的 `selectionKey` 恢复。只是勾选资料但未确认时仍是“可执行”。
2. 第 3 步只有在产物全部完整并点击“确认图谱输入”写入 `exportConfirmed=1` 后计为“已完成”；产物完整但未确认时仍是“可执行”。
3. 第 4 步在 `promptConfirmed=1` 时计为“已完成”；该确认态只存在于 URL，不写回服务端，刷新当前 URL 不应导致进度倒退。
4. 当资料集合、导出产物或索引前置条件变化导致确认态失效时，前端必须清理对应查询参数并重新计算进度；单纯轮询解析进度或 loader 重新加载解析状态，不应反向清理第 1 步确认态。

进度轨文案示例：

```text
已完成 3/6 · 50%
1 个步骤可执行 · 1 个步骤阻塞
```

执行中场景必须纳入计数渲染，例如：

```text
已完成 2/6 · 33%
1 个步骤执行中 · 1 个步骤可执行 · 2 个步骤阻塞
```

## 7. 步骤导航规则

URL 建议增加 `step` 查询参数：

```text
/app/knowledge-bases/7/build?materialIds=9,10&materialConfirmed=1&exportConfirmed=1&promptConfirmed=1&step=index
```

规则：

1. `step` 只决定当前查看哪一步，不代表该步骤可执行。
2. `materialIds` 表示小集合下已选择的资料集合，使用逗号分隔并在前端归一化为去重、稳定排序的 ID 列表，例如 `materialIds=9,10,12`。
3. 当 URL 同时存在 `selectionKey` 和 `materialIds` 时，`selectionKey` 优先：前端先按 `selectionKey` 从 `sessionStorage` 恢复稳定排序后的资料 ID 数组，并在下一次写 URL 时清理 `materialIds`；如果本地缺少该 `selectionKey` 对应数据，才降级读取 `materialIds`；如果两者都无法恢复有效资料集合，则定位回第 1 步重新选择。
4. `materialConfirmed=1` 表示用户已经点击“确认勾选”，只作为首轮前端向导确认态，不写回服务端；只有用户主动变更资料集合时才必须清理，例如修改 `materialIds`、从 `selectionKey` 恢复出不同资料集合、全选 / 清空选择或删除已选资料。解析状态轮询、资料解析状态变化或 loader 重新加载解析状态，不应清理 `materialConfirmed`；只有资料集合本身已失效时才清理。
5. `exportConfirmed=1` 表示用户已经点击“确认图谱输入”，只作为首轮前端向导确认态，不写回服务端；当任一导出产物缺失、资料集合变化或第 3 步重新执行导出后必须清理。
6. `promptConfirmed=1` 表示用户已经确认本次索引采用当前活动提示词，同样只作为首轮前端向导确认态，不写回服务端；当图谱输入产物变化、资料集合变化或第 3 步重新执行导出后必须清理。
7. 如果 `step` 缺失，默认定位到第一个“失败 / 执行中 / 可执行”步骤；如果全部完成，则定位到“问答效果验证”。
8. 如果 `step` 不存在，回退到默认定位规则，并清理非法 `step`。
9. 点击步骤轨和标题左侧图标回退都只改变 `step` 查询参数，并保留已有的 `materialIds`、`selectionKey`、`materialConfirmed`、`exportConfirmed`、`promptConfirmed` 等有效查询参数；只有非法 `step`、无效资料 ID 或失效确认态才按对应规则清理。
10. 主操作按钮仍由步骤状态和前置条件控制，不能通过跳转绕过阻塞。
11. 不再提供泛化的“下一步”按钮；底部主按钮必须使用当前步骤的业务动作文案，例如“确认勾选”“检查图谱输入”“导出缺失输入”“确认图谱输入”“开始构建索引”。
12. “上一步”从底部动作区移到当前步骤标题左侧，使用轻量 ghost 图标按钮表达，优先采用 `lucide-vue-next` 的 `ChevronLeft`，配合 tooltip / `aria-label` 展示“返回第 01 步：选择课程资料”等说明。
13. 第 1 步点击资料只切换选中 / 取消选中；真正进入下一阶段由“确认勾选”触发。
14. 已完成步骤默认不显示重复执行业务动作；需要重跑时以后通过次级动作或更多菜单承载。
15. 为兼容当前已存在的单选链接，首轮实现可以读取旧的 `materialId` 并转成单元素 `materialIds`；新写入 URL 时只写 `materialIds` 或 `selectionKey`，不再写新的 `materialId`。
16. 当前 Java `/api/v1` 的创建索引接口仍是知识库级 `POST /knowledge-bases/{id}/index-runs`，不接收资料 ID 列表。因此首轮资料集合是前端构建范围确认和解析 / 导出前置校验依据；如果后续要严格只索引所选资料，需要先由 Java 后端补充可审计的索引范围契约，再把资料集合传入索引运行。
17. 当前步骤标题区只保留步骤序号、标题、状态、标题图标回退和一句动态结论；不放长说明文案，避免把主舞台变成帮助文档。动态结论必须是业务结论，例如“已选 4 个资料，1 个待解析”，而不是操作教程。
18. 正常选择、全选、确认勾选这类预期内操作不显示蓝色提示条；提示条只用于失败、阻塞、运行中确认或用户必须处理的异常。
19. `materialIds` URL 直写只用于小集合：建议上限为 50 个资料或序列化后不超过 1200 字符。超过阈值时把稳定排序后的 ID 数组写入 `sessionStorage`，URL 改写为 `selectionKey=<hash>&selectionCount=<n>`；如果用户打开链接时本地缺少对应 `selectionKey` 且没有可降级读取的 `materialIds`，主舞台显示“资料选择已失效，请重新选择”，并定位回第 1 步。后续如需跨浏览器分享大批量选择，再补后端暂存选择集契约。
20. 用户点击阻塞步骤时仍渲染主舞台，不渲染空白：标题状态显示“未满足条件”，中部展示阻塞原因、缺失前置证据和“回到第 N 步处理”的路径，主操作禁用并在按钮下方显示禁用原因。

业务动作与回退规则：

1. 标题左侧图标回退：跳到前一个步骤，第一步隐藏或禁用；它只改变 `step`，不提交业务动作。图标按钮不显示文字，不使用厚重圆形边框，避免抢占标题视觉。
2. 步骤轨点击：只切换查看位置，不绕过前置条件。
3. 第 1 步“确认勾选”：没有资料时禁用；存在待解析 / 解析中 / 解析失败资料时进入第 2 步；全部已解析时直接进入第 3 步。
4. 第 2 步主操作：解析中时为“刷新解析进度”；存在失败或待解析资料时为“并行解析未完成资料”；全部完成时为“检查图谱输入”，点击后只进入第 3 步，不调用导出接口。
5. 第 3 步主操作：存在缺失导出产物且全部已解析时为“导出缺失输入”；全部完整但未确认时为“确认图谱输入”，确认成功后写入 `exportConfirmed=1` 并进入第 4 步。
6. 主操作成功后自动刷新 loader，并定位到下一个“可执行 / 执行中 / 需要用户确认”的步骤。

### 7.1 统一视觉规则

进度轨：

1. 六步横向节点应控制视觉密度：默认节点显示序号、状态图标和短标签；当前节点放大或加粗边框，并展开完整中文步骤名。
2. 非当前节点保持轻量尺寸，避免六个中文长标题在桌面端挤压；窄屏下进度轨横向滚动。
3. 节点 hover / focus 时通过 tooltip 展示完整步骤名、状态和阻塞原因。

紧凑摘要条：

1. 每一步的摘要条统一为横向 chip 组，例如 `[ 已选 4 个 ] [ 解析完成 3 ] [ 待处理 1 ] [ 导出缺失 0 ]`。
2. chip 颜色语义统一：成功绿、执行蓝、警告橙、失败红、中性灰；同一状态在进度轨、主舞台和诊断区保持一致。
3. 摘要条只放结果和计数，不放操作教程。

状态图标：

| 状态 | 图标 | 颜色 |
| --- | --- | --- |
| 已完成 | `CheckCircle2` | 绿色 |
| 执行中 | `Loader2`，旋转 | 蓝色 |
| 失败 | `XCircle` | 红色 |
| 未满足条件 | `Lock` | 灰色 |
| 可执行 | `Circle`，空心 | 主题色 |

主舞台与诊断区：

1. 主舞台使用白色背景、轻阴影和清晰边框，承载当前步骤的可操作内容。
2. 诊断区使用浅灰背景，例如 Element Plus `--el-fill-color-light`，不使用重阴影。
3. 主舞台和诊断区之间使用 8-12px 间距，不依赖粗分割线。

标题区：

1. 标题区最多两行：第一行放图标回退、步骤序号、步骤名称和状态徽标；第二行放一句 14px 次要色动态结论。
2. 动态结论必须是当前结果，例如“已选 4 个资料，1 个待解析”“索引运行 #15 已可用”，不能写成操作说明。

操作区：

1. 主操作按钮固定在主舞台底部动作区第一位，使用全宽样式，和内容区保持 16px 间距。
2. 主操作上方使用轻分割线或浅色背景区分内容与动作。
3. 刷新状态、查看日志等次级动作放在主按钮下方或右侧，视觉权重低于主操作。
4. 主操作禁用时，按钮下方显示单行禁用原因，例如“需要先完成资料解析”。

## 8. 主舞台内容设计

### 8.1 选择课程资料

主舞台应展示：

1. 当前知识库绑定课程。
2. 使用 `el-card` 承载资料选择区，内部使用 `el-table` 的 selection 列实现多选，避免卡片式列表在资料较多时密度不足。
3. 表格顶部工具条提供文件名搜索、解析状态筛选、导出状态筛选、全选当前筛选结果、清空选择。
4. 当前选中资料集合的摘要不再使用四个松散大块，而是收敛为紧凑摘要条：已选数量、待解析 / 解析中 / 解析失败数量、导出缺失数量。
5. “选择方式”这类使用说明不占据业务信息块，改为标题旁的小问号图标，鼠标悬浮时用 tooltip 展示“可多选；确认后如存在未完成解析会进入解析检查，否则进入导出检查”。
6. 不在资料表格下方追加“已选择 N 个资料，确认后...”这类蓝色说明条；选中状态直接由表格 selection、摘要条和主按钮状态表达。
7. 如果没有资料，显示“当前课程暂无资料”，并提供返回课程详情或资料管理入口。

交互：

1. 点击资料后切换该资料是否选中，并更新资料集合查询参数：小集合写 `materialIds`，大集合写 `selectionKey`。只要资料集合发生变化，就同时清理 `materialConfirmed`、`exportConfirmed`、`promptConfirmed` 等依赖当前资料集合的确认态。
2. 至少选中 1 个资料后步骤状态保持“可执行”；只有用户点击“确认勾选”且确认成功后，才把 `materialConfirmed=1` 写入 URL，并把第 1 步计为“已完成”。
3. “全选当前筛选结果”只选择当前筛选条件命中的资料；“清空选择”清空所有已选资料。
4. 点击“确认勾选”后，如果已选资料中存在待解析、解析中或解析失败资料，则进入“解析状态检查”。
5. 点击“确认勾选”后，如果已选资料全部解析完成，则跳过第 2 步，直接进入“导出图谱输入”。
6. 跳过第 2 步只表示当前选择不需要解析检查，不表示隐藏第 2 步；用户仍可通过进度轨回看。

### 8.2 解析状态检查

主舞台应展示：

1. 已选资料的解析状态聚合收敛为紧凑摘要条：总数、已完成、执行中、待处理 / 失败。
2. 下方使用任务列表展示每个已选资料，左侧为资料名称，中间为解析进度条和百分比，右侧为解析状态。
3. 每行可在展开或悬浮区域查看 MinerU 批次、最近更新时间、失败原因和解析产物数量，首屏不再使用四个松散信息块。

交互：

1. 未选择资料时阻塞，并提示先回到第一步。
2. 资料解析应按资料维度异步发起请求：对待解析 / 解析失败资料分别调用现有 `startParse(materialId)`，使用 `Promise.allSettled` 或等价队列并行提交，单个资料失败不阻断其他资料提交。
3. 每个资料拥有独立行状态：待处理、提交中、解析中、已完成、失败；行内进度条按轮询结果更新。
4. 待处理行显示 0% 灰色空进度条，不使用加载动画；提交中行使用短暂的蓝色不确定态条；解析中行显示真实百分比并可使用轻微条纹动效；失败行保留最后进度并使用红色状态；已完成行为 100% 绿色或主题成功态。
5. 存在解析中资料时主操作为“刷新解析进度”，并显示执行中状态。
6. 存在待解析或失败资料时主操作为“并行解析未完成资料”。
7. 所有已选资料解析完成时主操作变为“检查图谱输入”，点击后直接进入“导出图谱输入”，不在第 2 步发起导出。
8. 等待解析时显示“待处理”，不要和“解析中”混为一类。

### 8.3 导出图谱输入

主舞台应展示：

1. 使用与第 2 步一致的紧凑摘要条：已选资料数、产物完整资料数、缺失产物数、当前是否可确认图谱输入。
2. 下方按资料展示产物矩阵，左侧为资料名称，中间为三类必需产物状态，右侧为“完整 / 待导出 / 失败”。
3. 三类必需产物以短标签展示：
   - `标准化`：`graphrag_normalized_docs.json`
   - `章节输入`：`graphrag_section_docs.json`
   - `分页输入`：`graphrag_page_docs.json`
4. 不再使用“已选资料 / 导出完整 / 缺失产物 / 下游状态”四个松散信息块；用户需要先看见哪份资料缺什么。
5. 最近导出反馈只在失败、运行中或需要用户处理时显示；正常导出完成直接更新产物矩阵和主操作状态。

交互：

1. 任一已选资料解析中时阻塞，提示“等待解析完成后再导出”。
2. 任一已选资料解析失败时阻塞，展示第 2 步的失败原因，并建议回到“解析状态检查”处理。
3. 任一已选资料等待解析或未开始解析时阻塞，提示“需要先完成资料解析”。
4. 已选资料产物全部完整但尚未点击“确认图谱输入”时，步骤保持“可执行”；确认成功后写入前端确认态，步骤才变为“已完成”。
5. 存在缺失产物且所有已选资料解析完成时步骤为“可执行”，主操作为“导出缺失输入”。
6. 导出失败时在主舞台内展示错误，不挤到页面顶部。
7. 全部产物完整后主操作变为“确认图谱输入”，点击后写入 `exportConfirmed=1`、清理 `promptConfirmed`，并进入“提示词调优”。

### 8.4 提示词调优

主舞台应展示：

1. 这一页只回答一个问题：本次索引将使用哪套提示词策略。
2. 首轮使用一个策略确认面板展示“GraphRAG 当前活动提示词”，配合紧凑标签：图谱输入已导出、待确认 / 已确认、首轮可继续索引。
3. Java `/api/v1` 暂未暴露候选提示词、课程语境样本和评测结果，这些信息不占据首屏主区域，只在次要说明或后续增强入口中出现。
4. 后续有接口后再升级为候选列表：候选名称、评测结论、启用状态、失败样例入口。
5. 不再用多个信息块解释“数据来源 / 当前处理方式 / 候选与评测 / 本页确认”，避免用户误以为页面已经具备实时调优能力。

交互：

1. 图谱输入未导出时阻塞。
2. 在不新增后端接口的首轮中，图谱输入已确认后显示“可执行”，主操作为“确认提示词策略”。
3. “确认提示词策略”只表达用户已经看过本次索引将沿用当前活动提示词；点击后写入 `promptConfirmed=1`，不写回服务端，也不把候选详情伪装成实时数据。
4. 首轮不因为缺少提示词调优接口而阻塞“创建索引”；页面必须说明索引将沿用 GraphRAG 当前活动提示词。
5. 后续有 Java 接口后，已有可用调优结果时显示“已完成”，并展示当前候选名称与启用结论。
6. 后续有 Java 接口后，调优失败时在主舞台内展示失败原因，并阻止进入索引构建。
7. 这一步的目标不是让用户理解内部脚本细节，而是明确“索引将使用哪套课程语境提示词”。
8. 确认后主操作变为“进入创建索引”，刷新当前 URL 时仍能从 `promptConfirmed=1` 恢复完成态；如果第 3 步重新导出或资料集合变化，必须清理 `promptConfirmed`。

### 8.5 创建索引

主舞台应展示：

1. 使用紧凑摘要条展示：当前可用索引、最近运行、运行状态、是否可进入问答验证。
2. 下方使用索引运行行展示当前 / 最近运行：左侧为运行名称，中间为可用度进度条和百分比，右侧为“待构建 / 构建中 / 可用 / 失败”。
3. 运行详情只保留用户判断需要的信息：使用的图谱输入、提示词策略、失败原因和日志入口；开始 / 结束时间等次要信息放入展开区或侧栏证据。
4. 不再用四个卡片重复展示“最近运行 / 运行状态 / 可用索引 / 处理结果”，避免用户在状态之间来回比对。

交互：

1. 图谱输入不完整时阻塞。
2. 首轮不把提示词调优确认作为后端硬前置；页面只提示“将沿用当前活动提示词”。
3. 没有运行或最近运行失败时允许“开始构建索引”。
4. 最近运行处于“执行中”时主操作禁用，显示“正在构建，刷新状态”。
5. `POST /index-runs` 成功后刷新 `GET /knowledge-bases/{id}` 和 `GET /knowledge-bases/{id}/index-runs`，确认索引是否已经可用。
6. “索引成功但暂时不可用于问答”的判断必须基于当前 Java 字段：`latestIndexRunStatus === 'success'` 且存在 `latestIndexRunId`，但 `activeIndexRunId` 为空或 `String(activeIndexRunId) !== String(latestIndexRunId)`。当前后端通常在 `markSuccess` 后调用 `updateActiveIndexRunId`，这个中间态应只作为前端刷新 / 轮询保护，不应长期停留。
7. 触发中间态后，前端轮询 `GET /knowledge-bases/{id}` 和 `GET /knowledge-bases/{id}/index-runs`，建议间隔 2 秒、最多 5 次；仍未看到 `activeIndexRunId === latestIndexRunId` 时，第 5 步内部 `status` 继续保持 `running`，主操作改为“手动刷新”，摘要条用橙色警告 chip 显示“可用状态同步超时”，诊断区提供“查看索引日志”入口，并显示“索引已成功，但可用状态未同步，请刷新或查看索引日志”。不要把该状态归类为 `failed`，也不要把进度轨提前计为完成。
8. 有可用索引后主操作变为“进入问答验证”。

### 8.6 问答效果验证

主舞台应展示：

1. 使用紧凑摘要条展示：可用索引、验证模式、验证结论。
2. 主区域左右组织：左侧是验证问题，右侧是验证结果结论和回答摘要。
3. 验证成功后只展示用户最关心的结果：通过 / 未通过、回答摘要、会话详情入口。
4. 请求日志、原始响应和内部耗时不占据主舞台；需要排障时放到折叠诊断区或侧栏证据。
5. 缺少可用索引时不渲染空白结果卡，而是直接给出阻塞原因和回到第 5 步的路径。

交互：

1. 缺少可用索引时阻塞。
2. 有可用索引时允许发起问答验证。
3. 验证成功后显示回答摘要，并提供会话详情入口。
4. 验证失败时在主舞台内展示局部反馈。
5. 验证完成后主操作变为“重新验证效果”，方便用户在二次修订后快速复测。

## 12. 响应式规则

桌面端：

## 13. 验收标准

设计落地后至少满足：

1. `/app/knowledge-bases/:kbId/build` 首屏能看到整体进度条和当前步骤主舞台。
2. 当前步骤的状态、主操作、阻塞原因和证据不再藏在小块区域。
3. 当前步骤标题左侧有图标式回退按钮；底部不再出现泛化“上一步 / 下一步”，而是显示业务动作按钮。
4. 资料选择仍可通过 `materialIds` 或 `selectionKey` 恢复，并兼容旧的单个 `materialId` 链接。
5. 第 1 步资料选择使用表格多选，支持搜索、状态筛选、全选当前筛选结果、清空选择和问号 tooltip。
6. 只勾选资料时第 1 步保持“可执行”；点击“确认勾选”成功后写入 `materialConfirmed=1`，才计入已完成。
7. 点击“确认勾选”后，存在未完成解析资料时进入第 2 步；已选资料全部解析完成时直接进入第 3 步。
8. 大批量资料选择超过 URL 阈值时使用 `selectionKey` + `sessionStorage` 降级；当 `selectionKey` 与 `materialIds` 并存时优先恢复 `selectionKey`，缺少本地选择集时才降级读取 `materialIds`，两者都无效时回到第 1 步重新选择。
9. 第 2 步以资料解析任务列表展示，包含资料名、进度条、百分比和右侧解析状态；待处理、提交中、解析中、失败、已完成有不同进度条视觉。
10. 第 2 步支持并行提交多个资料解析请求，单个资料失败不阻断其他资料；全部完成后主操作变为“检查图谱输入”，点击只进入第 3 步。
11. 第 3 步以资料产物矩阵展示标准化、章节输入、分页输入是否完整；产物完整后仍需点击“确认图谱输入”才能进入第 4 步。
12. 第 4 步只展示本次索引将使用的提示词策略，不把未接入的候选与评测伪装成实时数据；`promptConfirmed=1` 能在刷新当前 URL 后恢复完成态。
13. 第 5 步以索引运行行展示当前 / 最近运行、可用度和可用索引；同步中和同步超时状态基于 `latestIndexRunStatus/latestIndexRunId/activeIndexRunId` 判断，超时后仍保持执行中状态并提供手动刷新和日志入口。
14. 第 6 步以验证问题 + 结果摘要组织问答效果验证，不展示冗余请求日志。
15. 阻塞步骤被点击时主舞台展示阻塞原因、缺失前置证据和回到前置步骤的路径，不渲染空白。
16. 进度轨、标题状态和诊断区复用同一套状态图标与颜色语义；摘要条使用统一 chip 视觉。
17. 主操作按钮全宽置于主舞台底部，禁用时显示单行禁用原因。
18. 索引和问答效果验证的失败反馈显示在当前步骤主舞台内。
19. 缺少可用索引时问答效果验证步骤阻塞，不能发起假验证。
20. `pnpm test` 覆盖步骤模型、进度计算、默认步骤定位、`selectionKey` 与 `materialIds` 并存优先级、`step` 查询参数保留 `materialIds/selectionKey/materialConfirmed/exportConfirmed/promptConfirmed`、旧 `materialId` 兼容、业务主操作映射、确认勾选条件跳转、解析任务行模型、结果面板模型、并行解析调度、索引同步超时状态，以及提示词调优在无后端接口时的首轮降级逻辑。
21. `pnpm test:e2e` 覆盖多资料选择后的查询参数恢复、标题左侧图标回退、确认勾选条件跳转、阻塞步骤主舞台、并行解析任务列表、产物矩阵、提示词确认刷新恢复、索引同步中 / 同步超时状态、索引失败反馈、缺少可用索引时的问答效果验证阻塞、问答效果验证失败在新主舞台中的局部反馈。
22. `pnpm build` 通过。
