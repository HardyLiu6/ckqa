# CKQA README 项目展示化改造设计

## 背景

当前根目录 `README.md` 已能准确说明 CKQA 的课程知识图谱问答定位、跨模块边界、技术栈与本地启动方式，但整体仍偏向开发部署文档。读者需要依靠文字自行想象学生端问答、管理端知识生产和异步问答链路，难以在短时间内判断项目是否已经形成真实产品闭环。

本次改造在不改变任何业务功能、接口契约和运行配置的前提下，将根 README 调整为“项目展示入口 + 开发运行入口”的双层文档，并加入来自当前真实系统的脱敏截图。

## 目标

1. 让读者在 README 首屏直接看到真实的学生端课程问答效果。
2. 用可维护的 Mermaid 图解释系统架构、知识生产、问答时序和查询模式关系。
3. 用管理端真实截图证明课程资料、构建流程、QA smoke 或问答运维能力已落地。
4. 增加“为什么不是传统 RAG”“Demo Walkthrough”“验证与评测”，突出 CKQA 的产品与工程差异。
5. 保留现有快速开始、仓库结构、开发文档、贡献约定和当前边界，避免 README 变成只有展示、无法落地的宣传页。

## 非目标

- 不修改 Java、Python 或前端业务代码。
- 不重新设计学生端或管理端 UI。
- 不为截图重新执行 PDF 解析或 GraphRAG 索引构建。
- 不使用 Mock API、伪造评测数字或提交尚无结果的空缺指标。
- 不把内部 Python `/v1` 描述为浏览器正式接口。
- 不同步重写 `README.en.md`；本轮只修复用户指定的中文根 README，英文入口后续单独对齐。

## 方案选择

采用“真实服务 + 现有完成态数据”方案：启动 Docker 基础设施、Java 后端、GraphRAG 服务和两套前端，复用已有课程、历史问答、知识库构建与运维记录。只有现有记录无法呈现学生问答核心效果时，才发起一次新的低成本问答，不重新建索引。

未采用以下方案：

- 全新演示流程：从上传、解析、建图到问答全部重跑，真实性高但耗时长，并会制造大量无必要运行态数据。
- 前端 Mock 截图：结果稳定，但不能证明 README 中描述的真实跨服务闭环。

## README 信息结构

改造后的中文 README 按以下顺序组织：

1. 项目标题、导航、定位与一期状态。
2. 学生端真实问答首图。
3. 核心能力表。
4. 为什么不是传统文档 RAG。
5. 知识生产链路 Mermaid 图。
6. 系统架构 Mermaid 图与 Java `/api/v1` 边界说明。
7. 问答链路 Mermaid 时序图。
8. 查询模式选择 Mermaid 图及 `smart` / `hybrid_v0` 语义说明。
9. 管理端真实能力截图。
10. Demo Walkthrough。
11. 验证与评测。
12. 技术栈、快速开始、仓库结构、开发文档、贡献约定与当前边界。

现有内容以重排和压缩为主，不重复堆叠同一事实。快速开始仍保留可复制命令，并继续强调密钥不可提交、浏览器只走 Java `/api/v1`。

## 图像资产

图片统一放入 `docs/assets/readme/`，使用稳定、语义明确的文件名：

| 文件 | 内容 | 最低要求 |
| --- | --- | --- |
| `student-qa-demo.png` | 学生端课程问答 | 能看到课程语境、问题、完整回答、查询模式、来源或检索轨迹中的至少两项 |
| `admin-materials.png` | 管理端课程资料或解析进度 | 能证明课程资料与解析状态来自真实业务页 |
| `admin-build-smoke-demo.png` | 构建流程、QA smoke 或问答运维 | 能证明索引构建后的验证或运维闭环 |

优先使用 1440×900 左右的桌面视口，截图裁切到关键内容，避免浏览器地址栏和大面积空白。若单页信息过长，可只截取可证明该能力的页面区域，不拼接伪造长图。

## 截图脱敏规则

截图使用现有本地课程和完成态业务记录，但公开资产不得出现：

- 真实姓名、用户名、邮箱、手机号、头像；
- JWT、Cookie、密钥、模型端点凭据；
- 课程内部 ID、会话 ID、任务 ID、构建 ID、资料对象路径；
- 本机绝对路径、数据库连接信息和仅供内部调试的错误堆栈。

采集时优先通过 Playwright 在页面加载后注入临时 CSS 或遮罩元素隐藏对应 DOM，再执行截图；不修改数据库中的原始值，也不为了脱敏改动产品代码。课程公开名称、教材章节名和正常问答内容可以保留。截图完成后必须人工查看原图，确认遮罩完整且没有破坏主要展示信息。

## Mermaid 图设计

### 知识生产链路

展示：课程 PDF → Java 上传入口 → MinerU 解析 → 标准化导出 → GraphRAG build run → LanceDB / 图谱产物 → QA smoke → 可问答知识库。图中区分 MySQL 元数据、MinIO 原文件和构建批次产物，突出其不是简单的 chunk + embedding。

### 系统架构

按四层表达：用户层（学生端/管理端）、业务编排层（Java `/api/v1`、认证、课程路由、SSE）、知识问答层（Python query-task、streaming、五种最终模式）、知识生产与存储层（PDF ingest、MinerU、MySQL、MinIO、LanceDB、Neo4j、模型网关）。

### 问答时序

展示：学生端提交问题 → Java 做课程/问答域与模式编排 → Python query-task 执行检索 → Java 桥接 `progress/delta/sources` → 浏览器流式渲染 → 最终消息与来源持久化。补充 `afterEventSeq` 断线续传语义，但不把瞬时 token 描述为数据库事实。

### 查询模式

明确 `smart` 是学生端推荐入口，最终落到 `basic`、`local`、`global`、`drift` 或 `hybrid_v0`。其中 `hybrid_v0` 表示 BM25 证据注入 GraphRAG Basic，不是 OpenAI 兼容模型名。

## 验证与评测内容

README 只展示能够在本轮复核的事实：

- 关键模块的自动化测试或构建命令；
- 当前路由评测集和断言口径（若能由测试资源与测试输出共同验证）；
- 已归档实验的指标（仅在报告文件仍存在且含明确时间、样本与口径时引用）；
- QA smoke 的能力说明与真实页面截图。

无法在当前仓库或运行环境中复核的成功率、平均延迟和引用率不写入 README。不同日期、课程或模型配置下会漂移的数据必须带上口径，不能作为无条件的系统承诺。

## 运行与采集流程

1. 确认 Docker Compose 中 MySQL、MinIO、One API、Neo4j 与 Redis 健康。
2. 启动 Java 后端；由其托管或复用 GraphRAG API。
3. 启动学生端与管理端 Vite 服务。
4. 使用本地演示账号登录，访问现有课程、历史问答、资料、构建与运维记录。
5. 通过临时 Playwright 采集逻辑设置固定视口、等待页面稳定、应用脱敏遮罩并输出 PNG。
6. 人工检查每张图片，再写入 README。
7. 运行文档与前端相关验证。

截图采集逻辑属于本次维护辅助，不默认新增常驻产品依赖。若需要保留脚本，则脚本必须通过命令行参数或环境变量接收基础 URL、账号和输出目录，不硬编码凭据或本机绝对路径。

## 验收标准

1. `README.md` 首屏包含真实、清晰、已脱敏的学生问答截图。
2. README 包含四张能在 GitHub 渲染的 Mermaid 图，且语义与当前代码边界一致。
3. 至少包含一张学生端截图和两张管理端关键能力截图。
4. 新增的“传统 RAG 对比、Demo Walkthrough、验证与评测”不含伪造能力或无依据数字。
5. 图片相对链接有效，文件体积适合仓库展示，截图中不存在个人信息、内部标识或秘密。
6. 保留并校准当前快速开始与开发文档入口。
7. `git diff --check` 通过。
8. `python scripts/audit_repo_drift.py --strict` 通过。
9. README 中 Mermaid 代码块和本地图片链接通过静态检查。

## 风险与降级策略

- GraphRAG 查询较慢：优先展示已完成历史问答；只有缺少合格记录时才新建一次问答。
- Java 托管 GraphRAG 启动失败：先独立启动 `graphrag_pipeline/utils/main.py`，不修改配置基线。
- 现有页面暴露过多内部标识：缩小截图区域并注入遮罩；若仍无法安全公开，则舍弃该页面，不能带敏感信息提交。
- 历史构建记录与当前代码不兼容：管理端截图改用资料列表、系统健康或问答运维等当前可读页面，不触发旧构建重放。
- Mermaid 在非 GitHub 阅读器中兼容性有限：保留清晰的标题、节点文本与正文解释，不额外维护易漂移的 PNG 流程图副本。
