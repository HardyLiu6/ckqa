# 标准化导出验证说明

本文档用于手工验证 `pdf_ingest` 的标准化导出结果，以及 `graphrag_pipeline`
对标准字段的保留情况。建议先按“单课程单 PDF 验证”跑通，再做批量抽样验收。

## 1. 验证目标

验证分为两部分：

1. `pdf_ingest` 是否输出了符合预期的 `normalized_docs.json` / `section_docs.json`
2. `graphrag_pipeline` 在同步到 `input/*.json` 时，是否保留了关键标准字段

重点关注：

1. 标题是否还带目录页码、点线或伪章节
2. 正文是否还残留 `[IMAGE]` / `[TABLE]` 占位符
3. 页码范围是否为 1-based 且合法
4. 长文是否被合理切分
5. `document_type`、`chapter`、`section`、`subsection`、`heading_level` 等字段是否保留

## 3. 单课程单 PDF 验证

以下示例使用课程 `os`、文件 `file_id=3`。如果你的课程有多份 PDF，建议始终带上 `--pdf-file-id` 或 `--file-id`。

### 第一步：重新导出标准化结果

在 `pdf_ingest` 目录下执行：

```bash
python scripts/pdf_processor/mineru_parser.py list
python scripts/pdf_processor/mineru_parser.py export-graphrag os --file-id 3 --mode section --with-page-docs --force
```

说明：

1. `--force` 用于覆盖旧的导出结果
2. `--with-page-docs` 会同时生成 `page_docs.json`，方便你做额外对照
3. 如果你想强行控制 chunk 粒度，可以额外加 `--max-chars 1800`

### 第二步：拉取 `normalized_docs.json` 到本地临时目录

在 `graphrag_pipeline` 目录下执行：

```bash
mkdir -p ./tmp_validate/os
python utils/fetch_from_minio.py os --pdf-file-id 3 --json-file normalized_docs.json --input-dir ./tmp_validate/os/normalized --clean
```

预期产物：

```text
graphrag_pipeline/tmp_validate/os/normalized/normalized_docs.json
```

说明：

1. 当前 `fetch_from_minio.py` 已支持直接拉取 `normalized_docs.json`
2. 标准文档会保留 `content` 字段，不会再被误判为空文档

### 第三步：对 `normalized_docs.json` 做抽样验收

在仓库根目录执行：

```bash
python pdf_ingest/scripts/pdf_processor/export_audit.py graphrag_pipeline/tmp_validate/os/normalized/normalized_docs.json --output /tmp/os_normalized_audit.json
```

重点查看输出中的：

1. `normalized_schema_error`
2. `invalid_page_range`
3. `noisy_heading`
4. `image_placeholder`
5. `table_placeholder`
6. `long_document`
7. `short_document`

推荐目标：

1. `normalized_schema_error = 0`
2. `invalid_page_range = 0`
3. `noisy_heading` 尽量接近 `0`
4. `image_placeholder = 0`
5. `table_placeholder = 0`

### 第四步：拉取 GraphRAG 实际使用的 `section_docs.json`

在 `graphrag_pipeline` 目录下执行：

```bash
python utils/fetch_from_minio.py os --pdf-file-id 3 --json-file section_docs.json --input-dir ./tmp_validate/os/section --clean
```

### 第五步：验证下游字段是否保留

在仓库根目录执行：

```bash
python - <<'PY'
import json
from pathlib import Path

path = Path("graphrag_pipeline/tmp_validate/os/section/section_docs.json")
data = json.loads(path.read_text(encoding="utf-8"))
sample = data[0]

for key in [
    "title",
    "text",
    "id",
    "course_id",
    "source_file",
    "document_type",
    "chapter",
    "section",
    "subsection",
    "heading_level",
    "heading_path",
    "heading_path_text",
    "page_start",
    "page_end",
]:
    print(f"{key}: {'OK' if key in sample else 'MISSING'}")
PY
```

预期：

1. `title`、`text` 必须存在
2. `document_type`、`chapter`、`section`、`heading_level`、`heading_path_text` 应该存在
3. `page_start`、`page_end` 应为正整数

### 第六步：对 `section_docs.json` 再做一次抽样验收

```bash
python pdf_ingest/scripts/pdf_processor/export_audit.py graphrag_pipeline/tmp_validate/os/section/section_docs.json --output /tmp/os_section_audit.json
```

这里主要关注：

1. `noisy_heading`
2. `invalid_page_range`
3. `image_placeholder`
4. `table_placeholder`
5. `long_document`

## 4. 可选：跑一次 GraphRAG 索引验证

如果你要确认“字段保留 + 内容规范化”之后，GraphRAG 还能正常消费：

在 `graphrag_pipeline` 目录下执行：

```bash
python utils/fetch_from_minio.py os --pdf-file-id 3 --clean
graphrag index --root .
```

建议检查：

1. `input/section_docs.json` 的字段是否完整
2. `output/documents.parquet`、`output/text_units.parquet` 是否更新
3. 索引过程是否因为异常 metadata 或空文本报错

## 5. 批量抽样验收

如果你已经有一批本地导出文件，可以直接对目录做扫描：

```bash
python pdf_ingest/scripts/pdf_processor/export_audit.py ./some_export_root --filename normalized_docs.json --filename section_docs.json --output /tmp/batch_export_audit.json
```

该命令会：

1. 递归扫描目录中的 `normalized_docs.json` 和 `section_docs.json`
2. 汇总每个文件的问题统计
3. 为每类问题保留少量抽样样本

适合以下场景：

1. 对一批课程做阶段性质量验收
2. 比较规则修改前后的问题分布变化
3. 快速定位哪门课的哪份导出最需要人工复核

## 6. 如何理解验收结果

### 可以视为严重问题

1. `normalized_schema_error`
2. `invalid_page_range`
3. `noisy_heading`
4. `image_placeholder`
5. `table_placeholder`

### 需要结合文档类型判断

1. `short_document`
2. `long_document`

说明：

1. `short_document` 在目录型文档、课程说明、章节引导页中可能少量出现
2. `long_document` 在附录、算法说明、图表密集章节中可能少量保留
3. 如果 `long_document` 很多，优先考虑重新导出时增加 `--max-chars`

## 7. 推荐验证顺序

建议按下面顺序执行：

1. `export-graphrag` 重新导出
2. 拉取 `normalized_docs.json`
3. 跑 `export_audit.py` 看标准化质量
4. 拉取 `section_docs.json`
5. 抽查字段保留
6. 再跑 `export_audit.py` 看下游投影质量
7. 如需要，再跑一次 `graphrag index`

## 8. 常见注意事项

1. 多 PDF 课程请始终显式带 `--file-id` / `--pdf-file-id`
2. 验收建议写到单独临时目录，不要直接覆盖正式 `input/`
3. `graphrag_pipeline/input/section_docs.json` 只有重新执行 `fetch_from_minio.py` 才会更新
4. `normalized_docs.json` 不是 GraphRAG 默认消费文件，主要用于验收与问题定位
