## Issue 001 — buildRun.status='interrupted' 枚举从未被后端使用

- **状态**：已规划
- **触发版本**：feature/prompt-confirmation-step（2026-05-14）
- **关联代码**：
  - `sql/ocqa.sql` — `knowledge_base_build_runs.status` enum
  - `sql/migrations/20260505_kb_build_runs.sql`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/index/dto/BuildRunUpdateRequest.java` — `@Pattern` 正则白名单
  - `frontend/apps/admin-app/src/views/pages/build-run-list-model.js` — `STATUS_LABELS`（已移除该 key）
  - `frontend/apps/admin-app/src/views/pages/module-content.js` — 状态过滤项（已移除）

### 背景

`knowledge_base_build_runs.status` 表定义了 6 个枚举：`pending / running / success / failed / interrupted / archived`。

后端代码从未真正写入过 `interrupted`，所有"中断 / 失败"路径（IO 异常、Interrupted 异常、超时、BusinessException）都通过 `markIndexFailed()` 统一标记为 `status='failed'`。

### 影响

- 数据库层 enum 定义保留 `interrupted` 占位，会让阅读 schema 的人误以为系统支持"被中断"语义
- DTO `BuildRunUpdateRequest` 正则白名单允许传入 `interrupted`，理论上前端 PATCH 可以构造一条无意义的 interrupted 记录
- 前端原本在过滤项 / `STATUS_LABELS` 里展示「已中断」选项，对用户没有实际意义且会造成困惑

### 当前处理（v0）

前端已经从 `STATUS_LABELS` 和过滤项里移除「已中断」选项，单测同步更新。后端枚举与 DTO 正则**保留不变**，避免引入 schema migration 带来的部署风险。

### 后续建议（v1）

按优先级分两步：

1. **（低成本）DTO 正则收紧**：把 `BuildRunUpdateRequest.status` 的 `@Pattern` 正则去掉 `interrupted`，避免外部 client 构造该值。同时检查所有 `getStatus()` 读取点是否对未知 status 有兜底
2. **（高成本）schema migration 去除 enum 值**：写一个 migration 把 `interrupted` 从 enum 中删除。需要先确认线上生产数据中**不存在** `status='interrupted'` 的行。MySQL 8.0 的 enum 修改需要 `ALTER TABLE ... MODIFY COLUMN`，建议在低峰期执行

### 触发条件

下次需要清理 `knowledge_base_build_runs` 表 schema 时一并处理。

## Issue 002 — qa_status 与 status 关系平级且未规范化

- **状态**：部分缓解，治理项仍待完善
- **触发版本**：feature/prompt-confirmation-step（2026-05-14）
- **关联代码**：
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/entity/KnowledgeBaseBuildRuns.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/index/KnowledgeBaseBuildRunService.java` — `runQaSmoke / syncQaSmokeTerminalIfAvailable`

### 背景

`knowledge_base_build_runs` 同时维护 `status`（流水线整体状态）和 `qa_status`（QA smoke 子状态）两个字段，前者覆盖整个流水线，后者只在 `current_stage='qa_smoke'` 时有意义。

`qa_status` 取值为 `pending / running / success / failed / skipped`，与 `status` 取值有部分重合但语义不完全对齐（例如 `qa_status='success'` 和 `status='success'` 都可能存在但触发时机不同）。

### 影响

- 阅读代码时容易误把 `qa_status` 当作 `status` 的一部分
- 前端展示「QA 状态」列时需要单独维护 `QA_STATUS_LABELS`，与 `STATUS_LABELS` 文案部分重复
- 在 `runQaSmoke()` 里需要同时维护 `status='running' + qa_status='running'`，不一致时会引发判定错乱

### 后续建议

可在下一轮"流水线状态机"重构时考虑：

- 把 `qa_status` 合并到 `build_metadata` JSON 里（按子流程命名空间隔离）
- 或者用统一的"阶段 × 子状态"表达，例如 `current_stage='qa_smoke'` + `current_stage_status='running'`，避免双字段平级

### 触发条件

修改 buildRun 状态机其他大改时一并处理。当前不影响功能，留作设计债。

## Issue 003 — 多轮上下文与学习记忆尚未形成成熟系统级闭环

- **状态**：已观察
- **触发版本**：context-memory-review（2026-06-03）
- **关联代码**：
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaContextAssembler.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaQuestionRewriteService.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/summary/QaSessionSummaryService.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/memory/QaMemoryContextService.java`
  - `graphrag_pipeline/utils/query_task_manager.py`
  - `graphrag_pipeline/utils/query_engine_history_poc.py`
  - `docs/superpowers/specs/2026-05-17-qa-context-session-management-design.md`

### 背景

当前 CKQA 已经具备一套最小多轮问答基础：

- MySQL 持久化 `qa_sessions` / `qa_messages` / `qa_retrieval_logs`
- Java 侧生成 `recent` / `summary_recent` / `summary` 上下文快照
- `basic / local / global / drift / hybrid_v0` 支持上下文快照、最近主题继承和追问改写，优先 LLM，失败后规则回退
- 会话摘要表和学习记忆表已经落库，`local` 模式可选注入最近历史与长期记忆
- Python `/v1/query-tasks` 已预留 `conversationHistory` 和实验性 `local_history` 查询策略

但对比成熟上下文管理系统，能力仍未闭环。参考资料：

- Claude Agent SDK 的 session 模型会自动持久化完整会话 transcript，并支持 continue / resume / fork：<https://code.claude.com/docs/en/agent-sdk/sessions>
- Claude Agent SDK cookbook 直接把 session transcript 作为会话侧栏、重命名、打标、fork 的数据源：<https://platform.claude.com/cookbook/claude-agent-sdk-05-building-a-session-browser>
- Claude Code 把项目指令、自动记忆、路径规则和技能分层加载，并说明哪些内容会在 compact 后重新注入：<https://code.claude.com/docs/en/memory>、<https://code.claude.com/docs/en/context-window>
- Anthropic 的 context management 公开材料把 context editing 与 memory tool 分成两类：一个清理陈旧上下文，一个把关键知识保存到窗口外：<https://claude.com/blog/context-management>

### 影响

- **上下文预算仍是字符级策略**：2026-06-03 已新增 `QaContextPolicy` 集中管理 6 条消息 / 1800 字 / 3500 字快照、记忆条数和历史预算，但仍缺少按模型 token、输入/输出预留、reasoning 预算和工具结果预算拆分的 reducer。
- **会话可恢复但不可分支回放**：系统有会话列表与消息表，但没有 transcript 级的可回放快照、fork/branch 语义、session 标签、重命名来源或“从某轮继续”的产品协议。
- **摘要是旁路增强，缺少 reducer 闭环**：`QaSessionSummaryService` 会生成滚动摘要，但缺少类似成熟系统的 reducer 策略矩阵、摘要质量评分、摘要漂移报警、保留/丢弃依据和强制保留系统约束规则。
- **学习记忆可注入但透明度不足**：学生端能开关和删除记忆，但当前响应不展示记忆正文、命中原因、过期策略、合并/去重记录，也不支持用户编辑或确认某条记忆是否应该继续参与后续问答。
- **模式覆盖已改善但语义消费仍不一致**：`basic / local / global / drift / hybrid_v0` 已支持 Java 侧上下文快照、最近主题继承和追问改写；但只有 `local` 会把最近历史/学习记忆转成 Python `local_history`，其他 GraphRAG 模式主要消费改写后的 `retrievalQuery` 和 Java 侧诊断字段。
- **可观测字段已落库但缺少系统化看板**：`qa_retrieval_logs` 已记录 context / rewrite / memory 字段，但缺少专门面向“多轮指代、主题切换、摘要质量、记忆注入”的运维聚合和回归评测矩阵。

### 建议方案

按低风险顺序拆分：

1. **上下文策略注册表**：已用 `QaContextPolicy` 集中收拢 `recent / summary_recent / summary / local_history` 的适用模式、窗口和字符预算；后续再把策略外置配置化，并在响应与运维日志里稳定输出策略 ID。
2. **token 预算与 reducer**：用 tokenizer 估算输入 token，把系统约束、课程范围、索引版本、当前问题、检索证据、历史摘要分别分配预算；摘要和最近消息由 reducer 统一裁剪。
3. **多轮评测集**：增加“1～5 轮连续指代”“主题切换后指代”“摘要后指代”“长期记忆冲突当前会话”的自动化样例，固定 retrieval query、rewrite method、history applied 和最终来源召回。
4. **记忆透明控制**：让学生能查看、编辑、停用、确认学习记忆；保存每条记忆的来源、最后命中时间、命中原因和过期策略。
5. **session transcript / fork**：在不破坏现有 MySQL 表的前提下，增加可回放 transcript 导出和“从某一轮新建分支会话”的最小协议。
6. **Python 双输入协议**：后续将 `retrievalQuery`、`generationContext`、`conversationHistory` 分开传给 Python，并明确每个 GraphRAG mode 是否消费 history。

### 触发条件

- 学生端正式上线前需要稳定支持连续追问、历史会话恢复和学习记忆透明控制
- 管理端需要复盘“为什么这一轮问答用了某段历史/记忆/摘要”
- 计划把 GraphRAG query engine history 从 PoC 升级为正式链路

### 2026-06-03 进展

- 已新增 `QaContextPolicy`，集中维护上下文窗口、字符预算、记忆预算、支持模式和 Python `queryEngineStrategy` 协议值。
- 已补充 Java/Python 离线回归测试，覆盖连续指代主题稳定、业务记忆策略名不再透传给 Python，以及 Python 侧继续拒绝非法 `queryEngineStrategy`。
- 学生端仍不暴露记忆正文；当前仅保留运维诊断字段。token reducer、transcript/fork、记忆编辑确认和摘要结构化主题仍是后续治理项。

### 2026-06-04 下一步优先建议

如果只从“提升学生体验和演示效果”选择下一项，优先做 **上下文理解可视化与演示回归 v1**，而不是先做 token reducer 或完整 transcript/fork。

原因：

- 语义主题栈 v1 已能支撑“连续追问、比较前者/后者、摘要窗口外恢复主题”的核心演示，但学生端目前很难看出系统到底理解了什么。
- `qa_retrieval_logs` 已记录 `resolvedTopic / topicSource / topicConfidence / rewriteMethod / contextStrategy / memoryApplied / topicEntityBindingStatus` 等字段，做可视化主要是产品呈现和少量接口整理，风险低于大改检索协议。
- 演示时最容易被追问的是“它为什么知道我说的它是谁”，安全展示“本轮理解为：死锁，来自最近对话/摘要/比较锚点”比继续堆内部规则更能建立信任。

建议最小版本：

1. 学生端 QA 回答区增加安全上下文提示：`已理解追问主题：死锁`、`来源：最近对话 / 摘要 / 比较前者`、`已参考最近对话`、`已使用学习记忆`。不展示学习记忆正文。
2. 管理端 QA 运维详情增加上下文诊断折叠区：`requestedMode / resolvedMode / resolvedTopic / topicSource / topicConfidence / rewriteMethod / contextStrategy / memoryApplied / topicEntityBindingStatus / historyFallbackReason`。
3. 增加一组可一键演示的 smoke 场景：连续 3 轮“它”、`死锁和饥饿...前者/后者...`、智能模式跨实际模式切换后追问。
4. 把上述场景纳入回归测试，固定学生端可见提示和运维字段，避免后续改上下文策略时演示链路退化。

## Issue 004 — 连续多轮“它”指代稳定性描述部分真实且 local_history 策略名未对齐 Python 协议

- **状态**：核心缺陷已修复；完整语义栈限制另见 Issue 005
- **触发版本**：context-memory-review（2026-06-03）
- **关联代码**：
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaContextAssembler.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaQuestionRewriteService.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaQuestionRewriteClient.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/memory/QaMemoryContextService.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/QaWorkflowService.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/QaTaskWorker.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/integration/graphrag/GraphRagTaskClient.java`
  - `graphrag_pipeline/utils/main.py`
  - `graphrag_pipeline/utils/query_task_manager.py`
  - `graphrag_pipeline/utils/query_engine_history_poc.py`

### 背景

审阅时待确认的问题描述为：

> 只在第一轮确定主题、之后连续 1～2 轮用“它”，系统大概率能正常解析。连续 3 轮以上用“它”，纯规则回退会失效，只能靠 LLM 路径在近窗里看到原始主题；一旦超过近窗（约 6 条消息或 1800 字）就要看摘要质量。

这个描述**方向真实，但细节不是严格代码事实**：

- 近窗上限确实存在：`QaContextAssembler` 固定最近 6 条消息、1800 字内容预算、3500 字快照预算。
- 规则回退确实只做低成本指代补全：命中“它/这个/该概念/前者/后者”等词后，把问题改写成 `关于上一轮主题「...」：原问题`。
- 规则回退的真实弱点不是“第 3 轮”这个硬阈值，而是 `latestCompletedTopic()` 会从最近一个完成的 user/assistant 对里抽取用户问题作为主题。连续追问时，最新用户问题本身可能就是“它和资源分配图有什么关系？”、“它怎么检测？”，于是规则主题会逐步退化成含“它”的追问短语，而不是最早的真实主题“死锁”。
- LLM 改写默认开启，但依赖 One API 可用、模型返回高置信度 JSON，且只能看到 Java 传入的快照/摘要；当快照缺少原始主题或摘要质量不足时，LLM 也没有可靠依据。
- `local` 模式另有一条长期记忆/最近历史注入路径，但它受学习记忆偏好、scope、预算和 Python `local_history` 可用性约束，不能视为所有模式的稳定兜底。

### 已观察到的额外协议风险

修复前 Java 与 Python 的 `queryEngineStrategy` 命名存在不一致：

1. `QaMemoryContextService` 返回业务策略名：`local_history_short_only`、`local_history_relevant_memory`、`local_history_preference_only`。
2. `QaWorkflowService` 在 memory 生效时把上述业务策略名写入 `QaRetrievalLogContext.queryEngineStrategy()`。
3. `QaRetrievalLogsServiceImpl` 原样持久化到 `qa_retrieval_logs.query_engine_strategy`。
4. `QaTaskWorker` 只要发现 `queryEngineStrategy` 或 `conversationHistory` 非空，就原样传给 `GraphRagTaskClient`。
5. Python `QueryTaskCreateRequest.queryEngineStrategy` 只允许 `cli` 或 `local_history`；`QueryTaskManager` 也只接受这两个值。

因此，学习记忆开启且 `local` 模式触发 history 注入时，Java 可能向 Python 发送 `local_history_preference_only` 等非法值，导致 `/v1/query-tasks` 422，或者使 history 策略无法进入预期的 `local_history` 路径。该核心风险已在 2026-06-03 修复。

### 2026-06-03 修复进展

- `memoryStrategy` 继续保留 `local_history_preference_only` 等业务诊断名；`queryEngineStrategy` 统一由 `QaContextPolicy` 映射为 Python 协议值 `local_history` 或空值。
- `QaTaskWorker` 对历史遗留的业务策略名做兼容归一化：只要 `local` 任务携带有效 `conversationHistory`，提交 Python 时使用 `local_history`。
- `QaTopicResolver` / `QaContextAssembler` 已跳过明显指代型用户问题，连续多轮“它”追问时继续继承最近一个非指代主题。
- 已补充回归测试：`QaContextAssemblerTest`、`QaWorkflowServiceTest`、`QaTaskWorkerTest`、`GraphRagTaskClientTest` 和 `graphrag_pipeline/tests/test_query_task_manager.py`。

### 2026-06-04 语义主题栈 v1 进展

- `basic / local / global / drift / hybrid_v0` 均支持上下文快照、最近主题继承和追问改写。
- 已新增 `QaTopicResolver` / `QaTopicStack` / `SessionSemanticState`，支持连续指代、`前者 / 后者`、简单比较主题、结构化摘要恢复。
- `qa_session_summaries` 和 `qa_retrieval_logs` 已保存 `latestTopic / activeTopicsJson / semanticStateJson` 等语义诊断字段。
- 已补充刁钻回归用例，覆盖乱序历史、摘要窗口外 `semanticStateJson` 恢复、比较锚点重置、显式新主题覆盖旧指代链。

### 影响

- 连续多轮“它”追问的规则主题退化已修复；超过近窗时可从结构化摘要主题/语义状态恢复，但仍依赖摘要字段质量。
- 当 LLM 改写服务不可用、置信度不足或 One API 配置异常时，规则回退已能继承非指代主题，但仍不具备实体级主题跟踪能力。
- `queryEngineStrategy` 协议不一致已修复；后续仍需观察 Python `local_history` PoC 到正式链路的稳定性。
- 运维日志里的 `memoryStrategy` 与 `queryEngineStrategy` 字段语义已拆分，后续可继续做聚合看板。

### 建议方案

1. **拆分字段语义**：已完成。`memoryStrategy` 保留 `local_history_preference_only` 等业务诊断名；`queryEngineStrategy` 只写 Python 协议允许的 `local_history`。

## Issue 005 — 可用语义主题栈 v1 仍不是完整语义栈

- **状态**：已观察
- **触发版本**：feature/session-semantic-state-v1（2026-06-04）
- **关联代码**：
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaTopicResolver.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaTopicStack.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/SessionSemanticState.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaContextAssembler.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaQuestionRewriteService.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/context/QaTopicEntityBindingService.java`
  - `backend/ckqa-back/src/main/java/org/ysu/ckqaback/qa/QaWorkflowService.java`

### 背景

2026-06-04 已完成“可用语义主题栈 v1”：

- 从摘要、结构化 `semanticStateJson`、最近历史和当前问题确定 `latestTopic`
- 支持“什么是 X / X 是什么 / 那 X 呢”等显式主题
- 支持连续“它 / 这个 / 那个 / 上述”等指代继承最近稳定主题
- 支持简单二元比较里的 `前者 / 后者`
- 支持摘要窗口外从结构化语义状态恢复最近主题和比较锚点
- 可选做 KG 主题实体弱绑定，但弱绑定只作为诊断增强，不作为强检索依赖

这能显著改善多轮追问体验，但它仍是“规则式主题锚点栈”，不是完整语义栈。

### 当前限制

- **规则解析覆盖有限**：当前主要识别显式定义句、`那 X 呢`、明显指代词和二元比较。复杂省略、隐式换题、`它们 / 两者 / 第二个 / 后两个`、三元以上比较、嵌套比较、跨段落并列主题仍可能解析不稳。
- **只有轻量主题锚点，没有完整实体状态**：`latestTopic` 是单主题字符串，`activeTopicsJson` 只保存主题列表和 `former/latter` 角色；尚未把主题稳定绑定到 KG entity id、实体类型、课程章节、关系路径或候选歧义集。
- **KG 弱绑定不是强依赖**：`QaTopicEntityBindingService` 可记录实体候选和耗时，但失败/超时会降级，不会改变 GraphRAG 检索策略，也不会保证回答一定围绕绑定实体。
- **摘要质量仍是上限**：结构化摘要字段能恢复主题，但系统还没有摘要质量评分、漂移检测、冲突修复、强制保留主题规则或“摘要不可信时回查 transcript”的 reducer。
- **学习记忆仍缺少语义冲突治理**：长期记忆只在 `local` 上下文注入路径里生效，并主要靠规则过滤；还没有 embedding/LLM 级相似度、冲突合并、过期策略解释和用户确认机制。
- **用户不可见导致信任感不足**：学生端目前看不到“本轮系统理解的主题是什么、来自最近对话还是摘要、是否使用了学习记忆”。演示时虽然后端字段可查，但用户侧缺少直观反馈。
- **缺少 transcript/fork 闭环**：已有 `qa_messages` 和摘要，但没有可回放 transcript 快照、从某轮 fork、新分支继承/丢弃语义状态的产品协议。

### 建议方案

按体验收益优先级拆分：

1. **上下文理解可视化与演示回归 v1**：优先做。学生端安全展示 `resolvedTopic / topicSource / rewriteApplied / contextApplied / memoryApplied`，管理端补充上下文诊断折叠区，并提供一键 smoke 场景。该项最能提升用户信任和现场演示效果，且不需要先大改检索协议。
2. **KG-grounded 主题状态 v1**：把 `latestTopic` 扩展为可选 `entityId / entityName / entityType / chapterHint / confidence / candidates`，仍保持 KG 绑定失败可降级。
3. **摘要 reducer 与质量门禁**：摘要生成时强制保存最近主题、比较锚点、保留/丢弃依据；摘要被用于指代恢复时记录 `summaryQuality / restoredFromSummary / fallbackReason`。
4. **长期记忆治理 v1**：补充记忆命中原因、过期策略、冲突合并和学生确认/停用入口；学生端默认只展示安全元数据，不展示敏感原文。
5. **transcript/fork 最小协议**：支持从某一轮复制消息、摘要和语义状态生成新会话，明确 fork 后是否继承学习记忆、KG 绑定和 active topics。

### 触发条件

- 需要向学生或评审演示“系统真的理解了连续追问里的它/前者/后者”
- 计划把语义主题用于更强检索 steering，而不仅是改写和诊断
- 后续要支持 session fork、摘要回放、跨 session 长期记忆治理

## Issue 006 —  邮箱验证登录与短信（手机号）验证登录

**状态**：邮箱验证登录已落地（2026-05-18）；手机号验证登录、第三方 OAuth 仍待规划。

### 当前进度

✅ **已落地（Phase A · 2026-05-18）**：
- 后端基础设施：Redis（infra/docker-compose.yml 新增 redis:7.4-alpine）+ spring-boot-starter-mail 依赖
- 配置：`CkqaSecurityProperties`（限频 / 邮箱验证码 / Turnstile）+ `CkqaEmailProperties`（mailer-type 切换 log/smtp）
- 失败限频：`LoginRateLimiter` 用 Redis 实现 5 次/15 分钟阈值 + 5 分钟锁定，覆盖密码登录与邮箱登录两类入口
- 人机验证：`TurnstileVerifier` 调用 Cloudflare verify endpoint，可通过 `ckqa.security.turnstile.enabled` 开关控制
- 邮箱验证码：`EmailCodeService` 用 Redis 存 6 位数字验证码（5 分钟 TTL），冷却 60 秒、每邮箱日额度 5 次
- 邮件发送：`LogEmailSender`（默认 mailer-type=log）+ `SmtpEmailSender`（mailer-type=smtp 时通过 spring-mail 真发送）
- 接口：`POST /auth/email/send-code` 与 `POST /auth/email/admin/login`，绕过 RBAC 的 `authenticated()` 守卫加入 SecurityConfig 白名单
- 前端：LoginView 加「账号密码 / 邮箱验证码」tab，60 秒倒计时发码，HumanCheck 组件支持 Turnstile widget 加载或开发态占位

❌ **仍未落地**：
- **真实邮件发送服务**：当前 `mailer-type=log`，验证码会打到后端日志而不是真的发邮件。上线前需要：
  1. 准备阿里云 / 腾讯云邮件推送 / 自建 SMTP 凭据
  2. 把 `application.properties` 中 `ckqa.email.mailer-type` 改为 `smtp` 并填 `smtp.host/port/username/password/ssl`
  3. 验证生产域名能正常发往主流邮箱（QQ / 163 / Gmail）
- **email/phone UNIQUE 索引**：当前是普通索引；上线邮箱登录前需要做数据清洗（去重 / 大小写归一 / 邮箱地址 trim）后再加 UNIQUE
- **Turnstile 真接入**：`ckqa.security.turnstile.enabled` 默认 false，开发态走 HumanCheck 占位条；上线时需要：
  1. Cloudflare Dashboard 申请 site key + secret key
  2. 后端配 `ckqa.security.turnstile.secret-key` 与 `enabled=true`
  3. 前端 `.env.production` 配 `VITE_TURNSTILE_SITE_KEY`
- **手机号短信验证码登录**（Phase B）
- **第三方 OAuth 登录**（Phase C，走 `auth_identities` 表）
- **forgot password 流程**：当前 LoginView「忘记密码？」按钮点击时会切换到邮箱验证码 tab，但没有专门的"重置密码"流程（用户只能用邮箱验证码登录，登录后到个人中心改密）

### 影响

- 暴力破解攻击已被 Redis 限频拦住，但缺少「锁定通知邮件」给真实账号主人
- 邮件链路仍依赖开发态日志，**上线前必须切到 SMTP**，否则验证码无法触达用户

### 后续 Phase 建议

**Phase B：短信（手机号）验证登录**
- `phone_verification_codes` 表 + 复用 `LoginRateLimiter` / `EmailCodeService` 模式
